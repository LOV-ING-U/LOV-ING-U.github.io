Spring 애플리케이션을 실행하면 맨날 보이는 HikariPool이 무엇일까?  
그리고 커넥션을 쓰고 반납한다 이런 표현을 많이 쓰는데 대체 내부적으로 어떻게 그것이 이루어지는 걸까?  
  
그리고 Shedlock 사용할 때 기본 생성자를 사용하지 않고 DB 시간으로 맞추기 위해 매개변수가 다른 생성자를 사용했었다(datasource를 사용하기 위해).  
  
DB와 datasource는 어떤 관계이고 HikariCP는 무엇일지 한번 정리해보자.  
  
## Connection Pool  
애플리케이션에서 DB에 접근하고, 연산하기 위해서는 어떤 형태로든 연결이 필요하다. 가만히 있으면 어떠한 연결이 되지 않으므로, 애플리케이션에서 하고자 하는 연산을 DB의 언어인 SQL로 변환할 수가 없고 작업도 할 수가 없다.  
  
Java 진영에는 이러한 DB Connection을 지원하는 API들이 있는데, 대표적으로 우리가 아는 JDBC나 JPA가 되겠다.  
  
Repository에서 JDBC나 JPA가 제공하는 JpaRepository와 같은 것들을 사용해서 코드를 짰던 경험이 바로 이 단계의 작업이다.  
  
그렇다면 **Connection Pool**은 추상적인 개념일까?  
  
## Spring App 시작시 무슨 일이 일어날까  
결론부터 말하면 추상적인 개념이 아니다. 결국 객체를 빌려 쓰는것 뿐이다.  
  
본인이 HikariCP를 사용하기로 했으면, DB와 연결시 **HikariDataSource**라는 구현체를 사용하게 된다.  
  
**HikariDataSource** 코드 주요 함수를 보면 아래와 같다.  
```java
public class HikariDataSource extends HikariConfig implements DataSource, Closeable {
    private static final Logger LOGGER = LoggerFactory.getLogger(HikariDataSource.class);
    private final AtomicBoolean isShutdown = new AtomicBoolean();
    private final HikariPool fastPathPool;
    private volatile HikariPool pool;

    // ...
    public Connection getConnection() throws SQLException { ... }
    public void close() { ... }

    // constructor
    public HikariDataSource(HikariConfig configuration) { ... }
    public HikariDataSource() { ... }
}
```  
  
즉 1개의 HikariDataSource 안에 실제로는 1개의 **HikariPool**이 존재하여, 내부 로직을 감추고 핵심 기능(getConnection, close 함수 등)만 노출시키는 구조이다.  
  
1) **Constructor** : HikariDataSource는 생성자가 2개 있는데, Configuration을 받는 버전은 미리 정한 설정값을 받는 방식이고, 매개변수가 없는 버전은 설정값 없이 일단 객체를 만드는 방식이다.  
  
의존성 주입을 할때도 Spring은 setter 주입과 같은 방식을 권장하지 않는다. 무결성을 보장하기 어렵기 때문인데, 왜 매개변수가 없는 기본 생성자를 지원하는 것일까?  
  
HikariCP는 Spring만 사용하는게 아니다. JVM 환경에서 모두 사용할 수 있는 라이브러리이므로 굳이 Spring 에서의 원칙을 지킬 필요가 없다. 그리고 설정값을 가져오는 경로가 설정 파일일수도 있고, 보안 서버일수도 있고 여러 상황이 있기 때문에 이러한 방식을 채택한 것으로 보인다.  
  
실제로 HikariConfig 구현을 뜯어보면,  
```java
public class HikariConfig implements HikariConfigMXBean {
  // sealed flag
  private volatile boolean sealed;

  // 커넥션이 아무 것도 안하고 있을때 그것을 유지할 수 있는 최대 시간
  private static final long IDLE_TIMEOUT;
  private volatile long idleTimeout;

  static {
    IDLE_TIMEOUT = TimeUnit.MINUTES.toMillis(10L);

    // 기타 등등 ...
  }
  
  // 기타 등등 ...
}
```  
가 있어 설정값 완료 즉시 sealed = true가 되고, 또한 HikariConfig 내부의 거의 모든 함수에서 sealed = true인지 체크하는 로직이 맨 앞단에 존재한다.  
  
따라서, 기본 생성자처럼 설정값을 나중에 채워도, connect 이전에만 이 값들을 전부 채우면 되고 그 이후에는 객체가 sealed 되므로 무결성에 문제가 없다.  
  
2) **DataSource** : DataSource는 커넥션 풀 관리 객체의 필수 함수(getConnection() 등)를 정의한 인터페이스이다. 해당 인터페이스 또한 **CommonDataSource**를 extend한 것인데, 여기에는 java.io를 이용한 logger 관련 함수 및 timeout 함수들 위주로 채워져있다. 즉 매우 일반적인 최소의 기능만 정의해놓은 인터페이스이다.  
  
3) **HikariConfig** : 설정값을 가져오는 보관소 역할이며, 일반적으로 **yaml 파일 또는 properties 등에서 설정값**을 읽어온다(이 경우, HikariConfig 내부의 **DataSourceProperties**가 설정값 대입을 도와준다). 또는 테스트코드 작성 등의 상황에서 직접 메서드로 값을 넣을 수도 있다(또는 properties 파일 경로를 제공해서 받아올 수도 있다). - **생성자를 확인해보면 보인다**.  
  
- **정리하면** : 애플리케이션이 구동될때 **HikariDataSource**가 만들어진다. HikariDataSource는 내부에 DB 커넥션 풀을 관리하는 HikariPool 객체를 가진다. 해당 객체는 설정값을 받을 수도 / 받지 않고 만들어질 수도 있으며, 일반적으로 설정값은 application.yaml (or .properties)와 같은 설정파일에서 가져온다.   
  
그럼 **HikariPool**은 어떤 역할을 하는지 알아보자.  
  
## HikariPool  
실제 코드를 뜯어보자. (원래 내부의 변수가 엄청나게 많은데, 핵심 부분만 추렸다)  
  
```java
public final class HikariPool extends ... implements ... {
  public static final int POOL_NORMAL = 0;
  public static final int POOL_SUSPENDED = 1;
  public static final int POOL_SHUTDOWN = 2;

  private final ConcurrentBag<PoolEntry> connectionBag;

  private final ThreadPoolExecutor addConnectionExecutor;

  private final ThreadPoolExecutor closeConnectionExecutor;

  private ScheduledFuture<?> houseKeeperTask;
  private final ScheduledExecutorService houseKeepingExecutorService;

  private final ProxyLeakTaskFactory leakTaskFactory;

  // get Connection 계열 함수
  public Connection getConnection() throws SQLException {
    return this.getConnection(this.connectionTimeout);
  }

  public Connection getConnection(long hardTimeout) throws SQLException { ... }

  // close Connection
  void closeConnection(PoolEntry poolEntry, String closureReason) { ... }

  // evict Connection
  public void evictConnection(Connection connection) { ... }
  private boolean softEvictConnection(PoolEntry poolEntry, String reason, boolean owner) { ... }

  // shutdown
  public synchronized void shutdown() throws InterruptedException { ... }

  // 이외 기타 등등...
}
```  
  
변수들 먼저 보고, 함수들을 살펴보자.  
### HikariPool 변수  
### POOL_NORMAL, POOL_SUSPENDED, POOL_SHUTDOWN  
커넥션 풀이 현재 어떤 상태인지를 int로 정의한 것이다. NORMAL = 정상 영업중, SUSPENDED = 잠깐 정지, SHUTDOWN = 커넥션 풀 종료(폐업) 을 의미한다.  
  
### connectionBag  
**PoolEntry**들의 Bag로 선언된다. 즉 실제 커넥션 풀들을 모아놓은 가방이다. HikariCP가 굉장히 빠른 커넥션 풀인데, 이 역할의 일등 공신이라고 해도 과언이 아니다.  
  
타입이 **`ConcurrentBag<PoolEntry>`** 인데, 중요한 만큼 하나씩 자세히 살펴보자.  
  
1) **PoolEntry**  
**DB와의 커넥션과, 그 커넥션이 커넥션 풀 내부에서 어떠한 상태인지를 나타내는 객체**이다. 내부 구현을 살펴보면 아래와 같다.  
  
```java
final class PoolEntry implements ... {
  Connection connection; // DB와 TCP/IP 연결이 완료된 DB 커넥션 객체
  private volatile int state = 0; // 해당 커넥션의 상태
  private volatile boolean evict; // 원래 다쓴 커넥션은 재활용을 위해 반납되는데, 얘가 켜져있으면 그냥 파괴

  // connection 반납(파괴 x, 재사용 o) -> 그래서 리턴 타입이 Connection인 것
  Connection close() { ... }

  // 각 state int가 무엇을 의미하는지 string으로 표시하는 함수. 자세한 사항은 실제 구현 참조
  private String stateToString() { ... }
}
```  
  
처음에 던졌던 '커넥션이 무엇인가?'에 대한 답이 나왔다. **커넥션은 PoolEntry Wrapper 클래스의 구성요소로, 그 안에는 실제 DB와 TCP/IP 연결이 완료된 DB 커넥션 객체가 담겨있고 이것이 커넥션이다**. 라고 말할 수 있을 것 같다.  
  
그리고 evict / state와 같은 변수들로 커넥션풀의 상태를 관리한다.  
  
2) **ConcurrentBag**  
쉽게 말하면 **커넥션 풀 내부의 커넥션들을, Lock 경합을 최소화하도록 구현한 HikariCP 전용 초고속 커넥션들 관리 주머니**이다. 내부 구현을 살펴보면 아래와 같다.  
  
```java
public class ConcurrentBag<...> extends ... {
  private final IBagStateListener listener;
  private final ThreadLocal<List<Object>> threadLocalList;
  private final CopyOnWriteArrayList<T> sharedList;
  private final SynchronousQueue<T> handoffQueue;

  // connection borrow
  public T borrow(...) throws InterruptedException { 
    List<Object> list = (List)this.threadLocalList.get();
    // ...

    for(T bagEntry : this.sharedList) { ... }
    // ...

    this.listener.addBagItem(waiting);
    // ...
  }
}
```  
  
커넥션을 가져오는 borrow 함수가 핵심인데, 총 3단계로 나뉜다.  
  
- **(1)** : **threadLocalList(내가 가지고 있는 객체)**를 뒤진다.  
- **(2)** : **sharedList(모든 스레드가 같이 사용하는 공용 주머니)**를 뒤진다.  
- **(3)** : **handoffQueue(커넥션을 받고 & 반납하는 정모 장소)**로 가서 반납되는 커넥션을 잡는다.  
  
즉 커넥션이 필요할 때 항상 본인 주머니부터 먼저 뒤지기 때문에, 커넥션 풀 할당이 굉장히 빠르다.  
  
문장이 좀 길어졌는데, **요약하면 connectionBag는 빠르게 커넥션들을 할당/관리하기 위한 특별한 바구니에(ConcurrentBag), 커넥션과 그 상태를 함께 나타내는 객체로 실제 DB 커넥션을 모델링(PoolEntry)하여 바구니에 넣은 것**이라고 표현할 수 있을 것 같다.  
  
### addConnectionExecutor  
DB와 새로운 연결(TCP/IP)을 맺는 역할이다. 커넥션이 부족할 경우, 사용자 스레드가 직접 DB 연결하는걸 기다린다면 응답 속도가 매우 느려질 것이다. 따라서, 이러한 작업은 비동기적으로 수행하여 사용자 스레드는 다른 일을 할 수 있게 된다.  
  
타입이 **ThreadPoolExecutor**이므로, **정해진 개수의 스레드를 미리 만들어두고, 이를 바탕으로 들어오는 작업들을 처리하는 도구**라는 의도로 addConnectionExecutor가 만들어졌음을 알 수 있다.  
  
즉, **DB와 새로운 연결을 맺을 때, 미리 만들어둔 스레드를 이용하여 비동기적으로 처리하는 실행기**라고 할 수 있겠다.  
  
### closeConnectionExecutor  
위의 addConnectionExecutor와 전반적인 컨셉은 같다. 다만 이건 실제 커넥션을 종료하는 즉 **TCP/IP 연결을 종료하는 실행기**이다.  
  
### houseKeeperTask & houseKeepingExecutorService  
**houseKeepingExecutorService**는 **houseKeeper**를 실행시키는 엔진이며, **houseKeeperTask**는 **houseKeeper**가 해야 하는 작업이다.  
  
갑자기 houseKeeper 라는 단어가 나왔는데, 이 친구는 **커넥션 풀의 커넥션을 관리(오래 방치된 커넥션을 정리하거나 부족한 커넥션을 채워넣는)하는 관리자**라고 이해하면 된다(HikariPool.class 내부의 private class로 정의되어 있다).  
  
코드 일부 핵심 로직만 살짝만 봐보면,  
```java
// HouseKeeper 클래스 내부 로직은 아니지만, run() 함수 내부에서 호출되는 HikariPool.class 내부의 함수이다.
private synchronized void fillPool(boolean isAfterAdd){
  // 1. 현재 커넥션 수가 max 수치보다 작으며 & 당장 할당 가능한(= 놀고 있는) 커넥션이 min 수치보다 작은가?
  // 그렇다면, 커넥션을 더 만들어야 한다.
  boolean shouldAdd = this.getTotalConnections() < this.config.getMaximumPoolSize() && idle < this.config.getMinimumIdle();

  // 2. 커넥션을 더 만들어야 하면(shouldAdd),
  if(shouldAdd) {
    // 1) min 수치에서 당장 할당 가능한 커넥션 수(idle)를 뺀 개수만큼 새로 만든다(countToAdd).
    int countToAdd = this.config.getMinimumIdle() - idle;

    // 2) 아까의 addConnectionExecutor로 비동기적 처리(커넥션 생성)
    for(...) { this.addConnectionExecutor.submit(...) }
  } else { print log }
}

private final class HouseKeeper implements Runnable {
  // 기타 등등 ...
  
  public void run() {
    // 1. 바뀐 설정값이 있는지 확인한 후 업데이트(동기화)
    // 이 줄 외에도 여러 설정값 동기화 로직 존재
    HikariPool.this.leakTaskFactory.updateLeakDetectionThreshold(HikariPool.this.config.getLeakDetectionThreshold());

    // 2. 시스템 시간이 바뀐 경우 처리
    // 1) 시간 동기화 오류 등으로 시스템 시간이 과거로 설정된 경우
    if (ClockSource.plusMillis(now, 128L) < ClockSource.plusMillis(this.previous, HikariPool.this.housekeepingPeriodMs)) {
      // 내부적으로 모든 커넥션을 soft-evict(현재 시간을 믿을 수 없으므로) 후 return
      // soft-evict에 관해서는 아래 함수 부분에서 서술할 것
    }

    // 2) starvation 등으로 이전(this.previous) 실행보다 주기의 1.5배 이상의 시간이 지난 경우
    // 즉 오랫동안 housekeeping이 실행되지 못한 경우
    if (now > ClockSource.plusMillis(this.previous, 3L * HikariPool.this.housekeepingPeriodMs / 2L)){
      // starvation 감지 log 출력
    }

    // 3. 놀고 있는 커넥션 정리
    // 커넥션 최소 개수와 최대 개수가 차이날 때만 동작(커넥션 개수가 고정이면 굳이 청소하지 않겠다)
    // idleTimeout은 아까 HikariConfig에 선언된 변수와 동일
    if (idleTimeout > 0L && HikariPool.this.config.getMinimumIdle() < HikariPool.this.config.getMaximumPoolSize()) {
      // connectionBag를 뒤져서 놀고 있는 커넥션 가져와서 처분(연결 종료)
    }

    // 4. 부족한 만큼 커넥션 재생성
    HikariPool.this.fillPool(true);
  }
}
```  
  
뭔가 되게 많은데, 정말 핵심 역할은 딱 이거다.  
**1. config 설정값 동기화 -> 2. 시스템 시간이 과거로 돌아간 경우 모든 커넥션을 evict 후 종료 -> 3. housekeeping 작업이 오랫동안 실행되지 않은 경우 경고 로그 출력 -> 4. connectionBag 뒤져서 놀고있는 커넥션 종료시킴 & 새 커넥션 생성**  
  
각 step의 동작에 대한 설명은 위 코드의 주석으로 간단히 핵심만 써놓았다(fillPool 동작도 위 코드에 주석으로 써놓았다. 간단한 동작이라 주석만 읽어봐도 이해가 될 것이다).  
  
즉 houseKeeper 덕분에 커넥션 풀의 수가 유지되고, 오래 방치된 커넥션을 정리하는 작업을 개발자가 일일이 신경쓸 필요가 없게 된다.  
  
(커넥션 풀은 우리 애플리케이션에서 일방적으로 맺고 끊기만 하는게 아니라 DB, 네트워크 등 다양한 장비/서버와 상호작용하므로 이들과 연결이 끊기는 것을 방지하기 위해 HikariCP는 주기적으로 새 커넥션으로 교체해주는 전략을 택한 듯 하다.)  
  
### leakTaskFactory  
**커넥션 누수를 탐지하는 실행기**이다. 마찬가지로 오래된 커넥션을 정리하는 역할을 하는데, 위의 houseKeeperTask와 다른 점은 **houseKeeperTask는 너무 오래 사용하지 않은 낡은 커넥션을 정리하지만, leakTaskFactory는 애플리케이션이 빌려가서 지나치게 오래 사용하고 있는 커넥션을 정리**하는 역할이다.  
  
leakTaskFactory는 **ProxyLeakTaskFactory** 타입의 객체인데, 상세 구현을 보면 아래와 같다.  
```java
class ProxyLeakTaskFactory {
    private ScheduledExecutorService executorService;

    // 커넥션 누수라고 판단하는 시간
    private long leakDetectionThreshold;

    ProxyLeakTaskFactory(long leakDetectionThreshold, ScheduledExecutorService executorService) {
        this.executorService = executorService;
        this.leakDetectionThreshold = leakDetectionThreshold;
    }

    ProxyLeakTask schedule(PoolEntry poolEntry) {
        return this.leakDetectionThreshold == 0L ? ProxyLeakTask.NO_LEAK : this.scheduleNewTask(poolEntry);
    }

    void updateLeakDetectionThreshold(long leakDetectionThreshold) {
        this.leakDetectionThreshold = leakDetectionThreshold;
    }

    private ProxyLeakTask scheduleNewTask(PoolEntry poolEntry) {
        ProxyLeakTask task = new ProxyLeakTask(poolEntry);
        task.schedule(this.executorService, this.leakDetectionThreshold);
        return task;
    }
}
```  
  
아무래도 해당 코드를 이해하기 위해서는 **ProxyLeakTask**를 먼저 이해해야 할 듯 하다. **ProxyLeakTask**의 코드를 보자.
